# WMS UI --> UX  文档

#### w一: UI和UX的区别

在传统的系统建设中，UI 往往被视为界面美观度的设计，例如颜色、按钮样式、排版等视觉呈现。但在如今的发展中，大多数B端系统则在规范性UI的前提下，侧重于UX，因为B端系统大多都是业务复杂度高、角色多、操作频繁，所以仅关注界面外观是不够的，因此我们选择从 UI → UX 的视角进行规范建设，更符合系统实际需求。原因如下：

1.  内部系统的核心价值不在“好看”，而在“好用”  
    WMS 的核心是提升公司内部的作业效率、减少操作失误、提升整体运营稳定性。UX 强调流程、路径、信息结构和交互成本，能够从“使用效率”层面真正支持业务目标，而 UI 仅能解决外观一致性的问题。
    
2.  目前WMS系统的涵盖饮片，颗粒剂和包材（下一期需求）的采购，仓储等操作，这些操作大多都是链路长、角色多，需要以流程为中心设计体验  
    仓库作业涉及收货、上架、拣货、盘点、包装、质检、调拨等多环节，不同角色（仓管员、操作员、主管）对界面需求不同。  
    UX 可以从整体流程出发设计规范，确保跨页面、跨角色的操作体验一致。
    
3.  减少错误、提升准确率是 UX 的范畴，而 UI 无法解决  
    现有的WMS 中，任何误操作（如错误上架、重复拣货、错扫码）都会造成直接成本。  
    UX 会从信息层级、点击路径、默认值、扫码模式、操作确认等方面设计交互来降低错误率，而 UI 并不关注这些关键要素。
    
4.  UX 强调可用性，符合仓储“高频高压操作场景”  
    仓库作业强调快速、高强度、高准确性，UX 会关注：
    
    *   单个操作步骤的最少点击数
        
    *   是否支持键盘/扫码枪无鼠标操作
        
    *   反馈是否实时清晰
        
    *   异常处理是否便捷  
        这正是 WMS 的实际需求。
        
5.  构建 UX 规范能提升整体系统一致性与可扩展性  
    UI 规范解决的是“视觉统一”；  
    UX 规范解决的是“行为、路径、反馈、流程的统一”。  
    当WMS后续扩展模块、构建二开应用时（比如升级为内部大型ERP），统一的 UX 规范能大幅降低开发和培训成本，提高项目交付效率。
    

总结一下: 

UI 管“好看”，UX 管“好用”; 个人认为B端的系统关键在于“好用”，因此大体的方向可以为在**完善UI的大框架的前提下，去建设和侧重UX**

#### 二: 现在为什么需要建立WMS的一套UI规范？

前面说了UI和UX的区别，那么为什么是现在去建立，是因为在开发的过程中，WMS之前是没有UI的，而UI有个相对主观的特性，每个人对界面美观、空间分布、色彩和风格都有不同的理解，这直接导致一个问题：任何人都可以对界面提出不同意见，而且每一种意见都“说得通”。但是我个人认为这种状态带来的影响又是系统性的:

1.  界面风格不统一，影响整体专业感
    
    在没有 UI 规范的情况下，不同模块可能出现：
    
    1.  颜色不一致
        
    2.  间距不一致
        
    3.  图标风格不一致
        
    4.  按钮位置、大小不一致
        
    5.  字体层次混乱
        
    
    这不仅影响美观性，也会让系统看起来像是由多个团队拼接出来的“混合体”，降低整体品牌感和专业性。
    
2.  设计沟通成本高，团队协作效率低
    
    比如：没有规范时，每次设计都需要团队内部重新讨论：这个按钮应该放左还是右？页面是分栏还是单列？用蓝色按钮还是绿色按钮？等等这些问题，其实到后面会发现，这种讨论本身不创造价值，却占用了大量研发时间，影响项目效率，如果有UI 规范后，这些讨论全部可以用规范去说明。
    
3.  功能迭代时可能容易出现视觉层面的“返工”
    
    同一个页面，由于没有统一的设计规则，需求迭代时可能产生以下情况：不同版本合并后风格冲突，新开发的页面与旧页面不一致，需要重新修整。现在已经有这种情况了，但是甘草现有的体系是以Base组件库去反推页面UI标准，其实是相反的，我们更应该是以UI标准去反推组件库的样式和风格
    
4.  新成员加入后难快速对齐设计风格
    
    这点，我写上去，是因为自己有点体会，刚加入的时候，因为没有UI，你现在能看到的一些新的页面，是属于个人美感导致的页面风格不统一，如果有规范，直接flow规范就行
    
5.  多端扩展（Web、PDA、移动）时更容易统一视觉体系
    

以现在的角度，其实这点虽然可能很难做到，因为协同以及工期的问题，但是还是要抛出来让大家思考一下，很多好的视觉体系建设，一定是多端协同的，一看到橙色，大家第一反应肯定是阿里系，一看到红色那就是京东系，看到蓝色可能大家会想到蚂蚁系，这就是视觉体系带来的潜移默化的品牌效应

就目前，我再次总结一句话：UI体系建设的目的，是想给大家减负，如果体系完善，以后有人对页面有不同的看法和想法，那就告诉他，我们前端和设计是有对这个的调研以及做法的合理性可以进行阐述的，同时也可以体现出专业性等问题

PS：下面是AI对我实现全fullDialog的评测:( 因为上次有同学问了这个问题，因为我写的时候是经验之谈，所以当时并没有思考自己为什么这么写，这也是我自己需要注意点的 ，不应该有路径依赖的问题 )

| 对比维度 | Ant Design Modal（官方） | PageDialog（createPortal 到 body） | 谁更适合 WMS/ERP 复杂场景？ |
| --- | --- | --- | --- |
| **渲染位置** | 渲染在当前组件树内（默认跟触发按钮同层） | 强制渲染到 `document.body` 最外层 | PageDialog 胜（无任何父级样式干扰） |
| **是否会被父容器裁剪** | 会被 `overflow: hidden`、`transform`、`fixed`、`z-index`、`filter` 等样式裁剪或遮挡 | 完全不会，永远在最顶层 | PageDialog 必胜 |
| **层级控制** | 多个 Modal 嵌套时需要手动调 `zIndex`（默认 1000），容易层级打架 | 天然最高层级，只需统一设置一个高 zIndex（如 1050）就永远在最上 | PageDialog 更稳 |
| **是否支持全屏/页面级布局** | 不友好：内容太多会出滚动条，内部放 Layout、Sider、Header 会乱套 | 天生支持：你可以直接在弹窗里放完整的 `<Layout><Header/><Sider/><Content/>`，像新页面一样布局 | PageDialog 完胜 |
| **是否保留父页面上下文** | 是（蒙层下父页面还在） | 是（但更彻底，因为内容区完全独立） | 平手，但 PageDialog 体验更好 |
| **返回时状态保留** | 关闭后父页面状态完全保留 | 完全一样 | 平手 |
| **是否支持从右侧滑入的全屏抽屉效果** | 可以（`width: 100vw` + 自定义动画），但会被父容器限制动画起点 | 丝滑无比，动画起点就是屏幕右侧，没有任何干扰 | PageDialog 胜 |
| **复杂表单/表格体验** | 小窗口感严重，表单 >30 个字段用户疯狂滚动，表格分页难受 | 像打开一个新页面一样宽敞，表格支持虚拟滚动、列固定、行内编辑都随意 | PageDialog 碾压 |
| **是否支持在弹窗里再嵌套 Modal/Drawer** | 容易层级冲突、蒙层穿透、滚动穿透问题 | 基本无问题（因为已经在 body 最外层） | PageDialog 更稳 |
| **移动端/PDA 适配** | 官方 Modal 在小屏下会强制居中变小，体验像手机弹窗 | 你可以自行控制为从底部上滑全屏（类似手机原生页面跳转），更适合 PDA 手持设备 | PageDialog 更灵活 |
| **破坏性变更风险** | AntD 升级时 Modal 样式/行为可能大改（v4 → v5 改过不少） | 你完全自己掌控，只用了原生 Portal，几乎零风险 | PageDialog 更安全 |
| **封装难度与灵活度** | 开箱即用，但想深度定制（全屏、自定义蒙层、复杂动画）需要各种 hack | 初始封装 20 行代码，之后想怎么玩都行（加过渡动画、自定义遮罩、键盘 ESC 等） | PageDialog 更灵活 |
| **性能** | 复杂内容多层嵌套时可能触发额外重排 | 直接在 body，性能反而更好 | PageDialog 略胜 |

其实从上面两种编辑可以看出来，我们更应该分不同情况去实施，面对复杂场景的表单交互，抽屉式的是一种弱打断、保持上下文的形态，可以看到列表的字段，以及可以快速修改字段，不影响整体观感，并且用区块去表达UI的整体结构，但是对于简单的字段编辑，大部分都是核心字段变更，同时也不需要区分区块的概念，这样的需求走弹窗更合适

下面是一些我认为的什么时候用抽屉，什么时候用弹窗的标准：

≤ 8 个字段： Modal（普通弹窗） 原因：状态变更、快速调整库存、单字段修改最快打开关闭，强打断但足够简单。

9~25 个字段：右侧抽屉  原因：弱打断，保留左侧列表上下文。

\> 25 个字段 或 多 Tab整页： FullDialog  原因：空间充足，表格随意展开，支持复杂的形式。
